# Codebase Concerns

**Analysis Date:** 2026-03-09

## Tech Debt

**Hardcoded absolute paths to remote training server:**
- Issue: Production training paths point to `/root/autodl-tmp/...` (AutoDL cloud GPU server). The codebase is designed to run on a specific remote machine, not a local environment. Paths are duplicated in multiple files and the code will fail silently on any other machine.
- Files:
  - `MultiTask_Stockformer_train.py` line 88: `tensorboard_folder = '/root/autodl-tmp/...'`
  - `MultiTask_Stockformer_train.py` lines 241-246: all `save_to_csv(...)` paths hardcoded
  - `lib/Multitask_Stockformer_utils.py` line 112: `path = '/root/autodl-tmp/...'` (Alpha_360 data directory hardcoded inside `StockDataset.__init__`)
  - `data_processing_script/stockformer_input_data_processing/results_data_processing.py` lines 35, 45-46
  - `data_processing_script/stockformer_input_data_processing/data_Interception.py` lines 26-30
  - `data_processing_script/stockformer_input_data_processing/Stockformer_data_preprocessing_script.py` lines 8, 53
  - `.vscode/launch.json`: `"program"` and `"cwd"` and `"python"` all point to `/root/...`
- Impact: Code is not reproducible on any machine except the original training server. Data path in `StockDataset` is not taken from config, making config-driven path management inconsistent.
- Fix approach: Replace all hardcoded paths with config file entries or environment variables. Alpha_360 path in `StockDataset` must be moved to the `[file]` section of `config/Multitask_Stock.conf`.

**Config-bypassed data path in StockDataset:**
- Issue: `StockDataset.__init__` in `lib/Multitask_Stockformer_utils.py` (line 112) loads the Alpha_360 bonus features from a hardcoded absolute path instead of reading from `args`. A commented-out alternative path is left in the code (line 111), indicating a previous date-range switch that was partially applied.
- Files: `lib/Multitask_Stockformer_utils.py` lines 111-112
- Impact: Changing the dataset date range requires modifying source code, not just config. The config parameter `alpha_360_dir` does not exist, so no config-driven switch is possible.
- Fix approach: Add an `alpha_360_dir` key to `[file]` section of `config/Multitask_Stock.conf` and pass it through `args` to `StockDataset`.

**Commented-out loss weighting logic:**
- Issue: The training loop in `MultiTask_Stockformer_train.py` (lines 295-307) contains a fully commented-out inverse-loss weighting scheme for multi-task loss balance. The current code uses unweighted sum `loss = loss_regress + loss_class`, which may under- or over-weight either task depending on magnitude. Similarly, lines 329-337 contain a commented-out accuracy-based model saving strategy, replaced by MAE-based saving.
- Files: `MultiTask_Stockformer_train.py` lines 295-307, 329-337
- Impact: Abandoned design decisions leave the codebase ambiguous about intended loss strategy. Accumulated commented code obscures the active logic.
- Fix approach: Delete commented-out blocks; document the chosen strategy in a comment or README.

**Dead code: `Stockformer_raw` class:**
- Issue: `Stockformermodel/Multitask_Stockformer_models.py` defines `Stockformer_raw` (lines 272-309), a predecessor architecture without multi-task heads. It is not imported or used anywhere in the active training script.
- Files: `Stockformermodel/Multitask_Stockformer_models.py` lines 272-309
- Impact: Inflates model file complexity; causes confusion about which class is the active model.
- Fix approach: Remove `Stockformer_raw` or move it to an archive module.

**Dead code: `graph_utils.py` has unused spectral graph functions:**
- Issue: `lib/graph_utils.py` defines `laplacian`, `largest_k_lamb`, and `get_eigv` (lines 13-31). These functions are never called; `loadGraph` ignores the spectral path (lines 34-37 show the `adj` and `graphwave` loading is commented out). The spectral graph wave approach was abandoned in favor of `struc2vec` GAT adjacency.
- Files: `lib/graph_utils.py` lines 13-31
- Impact: Dead code creates misleading impression of active spectral graph support.
- Fix approach: Remove unused functions from `graph_utils.py`.

**`.ipynb_checkpoints` committed to git:**
- Issue: Jupyter checkpoint files are committed: `.ipynb_checkpoints/MultiTask_Stockformer-checkpoint.py`, `.ipynb_checkpoints/MultiTask_Stockformer_train-checkpoint.py`, `.ipynb_checkpoints/Stockformer_train-checkpoint.py`, `Stockformermodel/.ipynb_checkpoints/`. These are auto-generated by Jupyter, not source files, and create confusion about the canonical code.
- Files: `.ipynb_checkpoints/` directory, `Stockformermodel/.ipynb_checkpoints/`
- Impact: Bloats repo, creates duplicate outdated code that may be mistaken for active source.
- Fix approach: Add `.ipynb_checkpoints/` to `.gitignore` and remove from version control.

**No `.gitignore` present:**
- Issue: There is no `.gitignore` file. As a result, `__pycache__/`, `.ipynb_checkpoints/`, `.DS_Store`, `venv/`, binary model checkpoints (`cpt/`), and TensorBoard event files (`runs/`) are tracked or untracked without policy.
- Impact: Risk of committing large binary files, IDE artifacts, or environment files in the future.
- Fix approach: Add `.gitignore` covering `__pycache__/`, `*.pyc`, `.ipynb_checkpoints/`, `venv/`, `.DS_Store`, `cpt/`, `runs/`, `log/`, `output/`.

**`requirements.txt` pinned to Linux platform with GPU-specific torch commented out:**
- Issue: `requirements.txt` opens with `# platform: linux-64`. The PyTorch dependency is commented out (`# torch==2.0.1+cu117`) meaning `pip install -r requirements.txt` does not install the core deep learning framework. Version `tqdm==4.62.3` is outdated relative to what is installed in `venv/` (4.x+).
- Files: `requirements.txt`
- Impact: Environment is not reproducible from the requirements file alone. New developers cannot set up a working environment without manual steps.
- Fix approach: Add an uncommented torch installation line with correct CUDA/CPU variant, or provide a `setup.sh` with the correct `pip install torch` invocation.

## Known Bugs

**`applymap` deprecated in pandas >= 2.0:**
- Symptoms: `results_data_processing.py` calls `data.astype(str).applymap(...)` which was renamed to `DataFrame.map()` in pandas 2.0 and removed in pandas 2.1+. The installed venv has pandas 2.3.3, so this call raises `AttributeError: 'DataFrame' object has no attribute 'applymap'`.
- Files: `data_processing_script/stockformer_input_data_processing/results_data_processing.py` line 15
- Trigger: Running `results_data_processing.py` against the installed venv.
- Workaround: Manually replace `applymap` with `map`; same change may be needed in `4_neutralization.py` if pandas is used there.

**`wape` metric computed but never returned or used:**
- Symptoms: In `lib/Multitask_Stockformer_utils.py` (lines 28-29), `wape` is computed inside the `metric()` function but the function returns only `(acc, mae, rmse, mape)`. The `wape` value is silently discarded.
- Files: `lib/Multitask_Stockformer_utils.py` lines 28-29, 44
- Trigger: Every call to `metric()` during validation and testing.
- Workaround: None required unless WAPE is needed; if intended as a metric, add it to the return tuple and callers.

**`StockDataset.__len__` uses `self.X` but class is also a `torch.utils.data.Dataset`:**
- Symptoms: The `StockDataset.__getitem__` method is defined and returns dicts but `MultiTask_Stockformer_train.py` does not use a `DataLoader`—it directly accesses `.XL`, `.XH`, etc. as numpy arrays. The `Dataset` interface (`__getitem__`, `__len__`) goes completely unused, so the Dataset class implementation provides no value and may mislead future developers.
- Files: `lib/Multitask_Stockformer_utils.py` lines 105-184, `MultiTask_Stockformer_train.py` lines 372-404

## Security Considerations

**External `sys.path.append` to unversioned remote dependency:**
- Risk: `Stockformer_data_preprocessing_script.py` (line 53) appends `/root/autodl-tmp/Stockformer/Stockformer_run/GraphEmbedding` to `sys.path` and then imports `ge` (graph embedding library) from it. This library is not in `requirements.txt`, not pinned to any version, and cannot be reproduced.
- Files: `data_processing_script/stockformer_input_data_processing/Stockformer_data_preprocessing_script.py` lines 53-55
- Current mitigation: None.
- Recommendations: Add `ge` (e.g., `graphembedding` package) to `requirements.txt` or document the specific version/source to install.

**No input validation on loaded numpy arrays:**
- Risk: All data loading (`np.load`, `pd.read_csv`) is done without validation of shape, dtype, or value ranges. Corrupted or mismatched input files would propagate silently and produce incorrect training results rather than failing with an informative error.
- Files: `lib/Multitask_Stockformer_utils.py` lines 109-121, `MultiTask_Stockformer_train.py` lines 377-406
- Current mitigation: None.
- Recommendations: Add assertion checks on loaded array shapes against config parameters (`T1`, `T2`, expected number of stocks).

## Performance Bottlenecks

**Alpha_360 bonus feature loading: O(n_files) serial CSV reads at dataset init:**
- Problem: `StockDataset.__init__` (lines 113-120) iterates over all CSV files in the Alpha_360 directory with a Python for-loop, reads each with `pd.read_csv`, and concatenates. This runs on every dataset instantiation (train, val, test = 3x) even though the source data is identical.
- Files: `lib/Multitask_Stockformer_utils.py` lines 113-120
- Cause: No caching; entire bonus array loaded from CSV three times per run.
- Improvement path: Pre-process and save `bonus_all` as a single `.npz` file during data preparation; load once and slice by train/val/test.

**`seq2instance` uses a Python for-loop for window slicing:**
- Problem: `seq2instance` and `bonus_seq2instance` (lines 166-184) create sliding windows using explicit Python loops over `num_sample` iterations. For large datasets this is slow compared to stride-trick numpy views.
- Files: `lib/Multitask_Stockformer_utils.py` lines 166-184
- Cause: Simple loop implementation.
- Improvement path: Replace with `numpy.lib.stride_tricks.sliding_window_view` or equivalent vectorized approach.

**`temporalEmbedding` uses Python for-loops over batch dimension:**
- Problem: `temporalEmbedding.forward` (lines 43-46 of `Stockformermodel/Multitask_Stockformer_models.py`) iterates over the batch dimension with `for i in range(TE.shape[0])` to apply `F.one_hot`, which prevents GPU vectorization for that step.
- Files: `Stockformermodel/Multitask_Stockformer_models.py` lines 43-46
- Cause: Misuse of `F.one_hot` which supports batched input natively.
- Improvement path: Replace loops with `F.one_hot(TE[..., 0].to(torch.int64) % 5, 5)` applied to the full batch at once.

## Fragile Areas

**`device` as a module-level global in model file:**
- Files: `Stockformermodel/Multitask_Stockformer_models.py` lines 275, 314, 356
- Why fragile: Three classes (`Stockformer_raw`, `StockformerBackbone`, `StockformerOutput`) all write to the same `global device` variable. The global is set only when a class is instantiated. Classes that use `device` before instantiation (e.g., `temporalEmbedding.forward` line 41-42, `temporalAttention.forward` line 150, `adaptiveFusion.forward` line 231) reference the module-level `device` directly without receiving it as an argument. If any layer is used before an outer model is constructed, it will fail with `NameError: name 'device' is not defined`.
- Safe modification: Pass `device` explicitly to all forward methods rather than relying on global state.
- Test coverage: No tests exist for these model components.

**`metric()` with suppressed numpy errors:**
- Files: `lib/Multitask_Stockformer_utils.py` lines 22-44
- Why fragile: The function wraps all computation inside `with np.errstate(divide='ignore', invalid='ignore')` and uses `np.nan_to_num` to replace NaN/Inf silently with 0. This means a completely degenerate prediction (all zeros, all identical) returns a valid-looking metric rather than indicating a problem.
- Safe modification: Add a warning or logging when the fraction of masked values exceeds a threshold (e.g., > 10% of labels are zero).

**`temporalConvNet` re-instantiates the same conv layers inside a loop:**
- Files: `Stockformermodel/Multitask_Stockformer_models.py` lines 183-191
- Why fragile: The `for i in range(levels)` loop assigns to `self.conv`, `self.chomp`, `self.relu`, `self.dropout` (overwriting each iteration), then appends the `nn.Sequential` to `layers`. Only the last iteration's `self.conv` etc. persists on `self`, yet `self.tcn` contains all levels correctly. This is confusing and could cause bugs if `self.conv` is referenced elsewhere; it is not a proper `nn.ModuleList`.
- Safe modification: Use local variables inside the loop rather than assigning to `self`.

## Scaling Limits

**Batch size 12 is hardcoded in config, not validated against GPU memory:**
- Current capacity: `batch_size = 12` in `config/Multitask_Stock.conf`. Works on the training GPU (AutoDL A100/V100 class).
- Limit: Reducing to a different GPU may require batch size tuning; there is no gradient accumulation fallback.
- Scaling path: Add gradient accumulation support or document minimum GPU memory requirement.

**Struc2Vec graph embedding scales O(n^2) with number of stocks:**
- Current capacity: Used for Chinese stock market (CSI 300, ~300 stocks).
- Limit: `Stockformer_data_preprocessing_script.py` generates a dense edge list (all pairs, O(n^2) edges) before fitting Struc2Vec. For universes >1000 stocks, this becomes very slow and memory-intensive.
- Scaling path: Apply edge threshold (e.g., keep only correlations above 0.5) before building the graph.

## Dependencies at Risk

**`pytorch-wavelets` (version 1.3.0) is not actively maintained:**
- Risk: The library has not received updates since 2021 and is not compatible with PyTorch >= 2.0 without patches. It is used as a core component of the DWT disentanglement step in `lib/Multitask_Stockformer_utils.py` (lines 9, 79-90).
- Impact: Upgrading PyTorch may break DWT disentanglement entirely.
- Migration plan: Either pin PyTorch to 2.0.x or reimplement the 1D DWT/IDWT using `torch.nn.functional` and `pywavelets` directly.

**Implicit `ge` (GraphEmbedding / Struc2Vec) dependency not declared:**
- Risk: The graph embedding library imported in `Stockformer_data_preprocessing_script.py` is not in `requirements.txt` and its source/version is unknown. If the remote server is decommissioned, this dependency is lost.
- Impact: Data preprocessing pipeline for generating `128_corr_struc2vec_adjgat.npy` cannot be reproduced.
- Migration plan: Pin the `ge` package version; add installation instructions to README or requirements.

## Missing Critical Features

**No data availability check at startup:**
- Problem: `MultiTask_Stockformer_train.py` loads data files at the top of `__main__` with no existence checks on the `.npz` files or Alpha_360 directory. If config paths are wrong or data is missing, the error is a raw Python exception from `np.load` or `os.listdir` with no guidance.
- Blocks: Cannot run training without manual debug of path errors.

**No early stopping:**
- Problem: Training always runs for the full `max_epoch = 100` configured epochs regardless of convergence. The learning rate scheduler (`ReduceLROnPlateau`) decays the LR to `min_lr=2e-6` but training continues unconditionally.
- Files: `MultiTask_Stockformer_train.py` lines 256-351
- Blocks: Wastes compute; final saved model may not be the best-generalizing checkpoint.

## Test Coverage Gaps

**No test suite of any kind:**
- What's not tested: All model components (`Stockformermodel/`), all utility functions (`lib/`), all data processing scripts (`data_processing_script/`), and the full training loop (`MultiTask_Stockformer_train.py`).
- Files: Every `.py` file in the project.
- Risk: Silent regressions in data preprocessing, metric computation, or model forward pass changes are undetectable without manual re-running of the full training pipeline.
- Priority: High — at minimum, unit tests for `metric()`, `disentangle()`, `StockDataset`, and `masked_mae()` would protect the most critical computation paths.
